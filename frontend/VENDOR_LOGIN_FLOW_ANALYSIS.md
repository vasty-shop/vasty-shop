# Vendor Login Flow Analysis: Complete Breakdown

## Critical Issue Found: JWT Payload Structure Mismatch

### The Problem

There is a **CRITICAL MISMATCH** between:
1. How the backend generates vendor JWT tokens
2. What the JWT strategy expects to validate
3. How controllers access the user ID

## The Issue Cascade

### What the token CONTAINS (generated by backend):
```json
{
  "userId": "abc123",
  "email": "vendor@example.com",
  "role": "vendor",
  "shopId": "shop123"
}
```

### What the JWT strategy EXPECTS:
```json
{
  "sub": "abc123",
  "email": "vendor@example.com",
  "role": "vendor"
}
```

### What gets attached to req.user:
```json
{
  "id": undefined,        // ❌ Because payload.sub doesn't exist
  "email": "vendor@example.com",
  "role": "vendor",
  "userId": undefined     // ❌ Not set by JWT strategy
}
```

### What controllers try to access:
```typescript
req.user.userId  // ❌ UNDEFINED!
```

---

## Why Authentication Appears to Work But Requests Fail

1. **Login succeeds** - Token is generated and returned
2. **Token is stored** - localStorage has valid JWT
3. **Token is sent** - Authorization header is added to requests
4. **Token is validated** - JWT signature is valid, not expired
5. **JWT strategy runs** - But `payload.sub` is undefined
6. **User validation fails or returns null** - Can't validate user with undefined ID
7. **Controllers receive invalid req.user** - `req.user.userId` is undefined
8. **All protected endpoints fail** - Can't query database with undefined user ID

---

## The Fix: 3 Options

### Option 1: Fix Token Generation (RECOMMENDED)

Change `generateVendorToken()` to use `sub` instead of `userId`:

```typescript
// File: backend/src/modules/auth/auth.service.ts (Line 359)
private generateVendorToken(userId: string, email: string, shopId: string): string {
  const payload = {
    sub: userId,      // ✅ Change userId to sub
    email,
    role: 'vendor',
    shopId,
  };
  return this.jwtService.sign(payload);
}
```

Also fix regular `generateToken()`:

```typescript
// File: backend/src/modules/auth/auth.service.ts (Line 115)
private generateToken(userId: string, email: string, role: string): string {
  const payload = {
    sub: userId,      // ✅ Change userId to sub
    email,
    role
  };
  return this.jwtService.sign(payload);
}
```

**Why this is best:** Follows JWT standard convention (`sub` is standard for subject/user ID)

---

### Option 2: Fix JWT Strategy

Change JWT strategy to use `userId` instead of `sub`:

```typescript
// File: backend/src/modules/auth/strategies/jwt.strategy.ts
export interface JwtPayload {
  userId: string;   // ✅ Change sub to userId
  email: string;
  role?: string;
  shopId?: string;
}

async validate(payload: JwtPayload) {
  const user = await this.authService.validateUserById(payload.userId);

  if (!user) {
    throw new UnauthorizedException('User not found');
  }

  return {
    userId: payload.userId,  // ✅ Return userId
    id: payload.userId,      // Also keep id for compatibility
    email: payload.email,
    role: payload.role || user.role,
    shopId: payload.shopId,
    ...user,
  };
}
```

**Downside:** Non-standard JWT structure

---

### Option 3: Fix Both (Most Comprehensive)

Use `sub` in token generation AND ensure JWT strategy properly maps it:

```typescript
// Token generation: Use sub
private generateVendorToken(userId: string, email: string, shopId: string): string {
  const payload = {
    sub: userId,
    email,
    role: 'vendor',
    shopId,
  };
  return this.jwtService.sign(payload);
}

// JWT Strategy: Map sub to userId for controllers
async validate(payload: JwtPayload) {
  const user = await this.authService.validateUserById(payload.sub);

  if (!user) {
    throw new UnauthorizedException('User not found');
  }

  return {
    userId: payload.sub,    // ✅ Map sub to userId
    id: payload.sub,
    email: payload.email,
    role: payload.role || user.role,
    shopId: payload.shopId,
    ...user,
  };
}
```

**Why this is best:** Standard JWT + backwards compatibility
